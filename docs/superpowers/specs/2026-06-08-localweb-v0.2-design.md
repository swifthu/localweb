# localweb v0.2 Design — UI 现代化 + Dashboard + 深度识别

**Date:** 2026-06-08
**Status:** Approved (brainstorming complete)
**Author:** jimmyhu
**Supersedes (in part):** `2026-06-08-localweb-design.md` §3 (Non-Goals — front-end framework prohibition remains; this spec only *extends* what's there)

---

## 1. Purpose

在 v0.1 已交付的"本地端口管理 Web 面板"基础上,v0.2 解决三个用户痛点:

1. **UI 太简陋** — 当前 style.css 39 行,无设计感;改成现代化深色/浅色双主题、CSS variables 系统、Grid/Flexbox 布局
2. **缺 Dashboard 总览** — 当前只有"服务列表"一种视图,加一个顶部 Dashboard 区(总览统计 + 服务分类汇总)
3. **服务识别不够深** — 当前 detector 只识别 label(command 名)和 HTTP 头,看不出"占用端口的是什么软件";扩展为同时采集 exe 路径、启动时间、PPID,并通过内置端口预设库(5432→PostgreSQL 等)直接标出

## 2. Goals (本期)

- Dashboard 默认显示:总端口数、按协议/按应用的分类汇总、最近 1 小时事件数
- 每个服务卡片显示:label(高置信度识别名)、进程名、exe 路径、启动时间、协议徽章、端口预设匹配
- 进程分组:同 exe basename 的多个端口聚成一组,可折叠/展开
- 双主题:深色(默认)+ 浅色,自动跟系统,手动 toggle,localStorage 持久化
- 保持"无构建步骤,原生 HTML/JS"原则;前端按职责拆 ES Modules
- 后端 API 兼容 v0.1(老客户端能跑,新字段是 additive)

## 3. Non-Goals (本期不做)

- macOS 专属增强(应用图标、bundle ID、codesign 签名)— 跨平台承诺
- 端口预设社区贡献 / 在线数据库同步
- 历史数据存储 / 趋势图(用户已确认不做)
- 响应式移动端布局(本工具桌面场景使用)
- i18n 国际化
- 引入任何前端框架(React/Vue/preact)— 保持原生 ES Modules
- 用户自定义预设(`~/.config/localweb/presets.yaml` 留作 v0.3 钩子)

## 4. Tech Stack(增量,基于 v0.1)

| 层 | v0.1 | v0.2 增量 | 理由 |
|---|---|---|---|
| 后端运行时 | Node 26 + TS strict | 不变 | — |
| 后端模块 | 见 spec §5 | 新增 `procinfo.ts` / `presets.ts` | 深度识别需要 |
| 前端打包 | 无构建 | 不变 | 保持承诺 |
| 前端结构 | 单 `app.js` 275 行 | 拆 5 个 ES Modules + `app.js` bootstrap | 职责分离,可维护性 |
| 主题 | 硬编码深色 | CSS variables + `data-theme` + 系统检测 | 现代化基础 |
| 布局 | Flexbox 简单流 | CSS Grid 卡片墙 + 自适应断点 | Dashboard 需要 |
| 测试 | Vitest | 增量 `procinfo.test.ts` / `presets.test.ts` | 新模块 |
| HTTP / WS | Express + ws | 不变 | v0.1 兼容 |

## 5. 模块扩展(相对 v0.1)

### 5.1 后端新增模块

```
src/server/
├── procinfo.ts       # 新增:exe 路径 + 启动时间 + PPID 采集(跨平台)
├── presets.ts        # 新增:端口→服务名/图标/颜色 预设库(纯数据)
├── types.ts          # 扩展 Service 新增 5 字段
├── scanner.ts        # 扩展:Scanner 在 enrich 阶段调用 procinfo
├── detector.ts       # 扩展:接收 procinfo 额外信息,做更准的 label
├── routes/services.ts # 不变(返回的 Service 自然带新字段)
└── ... (其余不变)
```

### 5.2 前端组件拆分(原生 ES Modules,无构建)

```
src/public/
├── index.html        # 改:加 Dashboard 区块、theme toggle、search 框
├── style.css         # 改:CSS variables 主题系统 + Grid/Flexbox 现代化布局
├── theme.css         # 新增:CSS variables 主题定义(dark/light 两套)
├── app.js            # 改:bootstrap + WS 接线,各功能委托给组件
└── components/
    ├── dashboard.js  # 总览统计 + 服务分类汇总(可独立渲染)
    ├── services.js   # 列表渲染、分组折叠、卡片
    ├── preshared.js  # 预启动面板(从 app.js 提取)
    ├── dialog.js     # 通用确认对话框(从 app.js 提取)
    ├── theme.js      # 主题切换 + localStorage + system preference 检测
    └── filters.js    # 协议筛选 + 搜索框(从 app.js 提取)
```

每个组件文件目标 `< 150 行`,导出 `init(ctx)` / `render(state)` 等简单 API。

### 5.3 中央状态(在 `app.js` 顶层)

```javascript
const state = {
  services: new Map(),         // pid → Service
  preshared: new Map(),        // name → Preshared
  filter: { tcp: true, udp: false, search: "" },
  groupByExe: true,            // 同 exe basename 折叠
  theme: "dark",               // "dark" | "light"
};
```

各组件订阅 `state` 变化(简单 pub-sub),调用各自 `render()`。

## 6. 数据流(相对 v0.1 增量)

### 6.1 启动序列(不变)

```
localweb CLI → index.ts → config → port → http → ws → preshared → scanner(2s tick)
```

### 6.2 扫描循环扩展

```
scanner.ts tick(每 2s):
  ├─ runLsof() → rawPorts
  ├─ procinfo(pid) → { exePath, startedAt, ppid }   ← 新增,模块级 cache
  ├─ detector.enrich(rawPort, procinfo)             ← 扩展,接收 procinfo
  │     ├─ 命令行 + 头识别 → label (v0.1)
  │     └─ presets.lookup(port) → servicePreset?    ← 新增
  ├─ groupKey = basename(exePath) || command         ← 新增
  └─ 推送 Service(含新 5 字段) 到 WS
```

### 6.3 端口预设匹配(`presets.ts`)

```
内置 Map<port, Preset>:
  5432 → { name: "PostgreSQL", icon: "elephant", color: "#336791" }
  3306 → { name: "MySQL",      icon: "dolphin",  color: "#00758f" }
  6379 → { name: "Redis",      icon: "stack",    color: "#dc382d" }
  27017 → { name: "MongoDB",   icon: "leaf",     color: "#00684a" }
  80, 443, 8080, 8443 → "HTTP/HTTPS Proxy"
  3000, 5173 → "Vite dev server"
  ... (约 30+ 条)

匹配规则:精确匹配 port;无匹配返回 null
```

### 6.4 进程组(同 exe basename)

```
groupKey 计算:
  exePath = "/Applications/Spotify.app/Contents/MacOS/Spotify"
  basename = "Spotify"
  → groupKey = "Spotify"  (主键)

同 groupKey 的多个 pid 聚成一组,UI 折叠为单卡
点 [展开] → 显示该 group 下所有 pid+port 列表
```

### 6.5 前端渲染

```
state.services 更新(WS added/removed/updated)
  ↓
app.js 触发 pub-sub
  ├─ dashboard.render(state)        # 重算总览
  └─ services.render(state)         # diff 列表 DOM(按 groupKey 折叠)
  ↓
preshared-update 走 preshared.render(state)
filter/search 变化 → filters.apply(state) → services.render
theme toggle → theme.apply(state) → <html data-theme="...">
```

### 6.6 主题切换

```
启动时:
  1. localStorage.getItem("localweb-theme") ? 优先
  2. window.matchMedia("(prefers-color-scheme: light)") 跟随系统
  3. 默认 "dark"

切换时:
  1. <html data-theme="dark|light"> 触发 CSS variables 切换
  2. localStorage.setItem("localweb-theme", newValue)
  3. 触发 pub-sub 让需要重渲染的组件重新计算颜色
```

## 7. REST + WS API

### 7.1 REST(完全兼容 v0.1,以下为新增)

| Method | Path | Body | 响应 | 状态 |
|---|---|---|---|---|
| GET | `/api/services` | — | `Service[]`(新字段为可选) | v0.1 兼容,新字段 additive |
| GET | `/api/presets` | — | `Record<port, Preset>` | **新增**,前端初始化时拿 |
| 其他 | — | — | — | 不变 |

### 7.2 WebSocket(不变)

消息类型与 v0.1 100% 相同。`Service` 对象在推送中多了 5 个可选字段。

### 7.3 类型扩展

```typescript
// types.ts: Service 扩展
interface Service {
  // ... 现有 11 字段(v0.1)
  pid: number;
  port: number;
  protocol: "tcp" | "udp";
  address: string;
  command: string;
  cwd?: string;
  user: string;
  label: string;
  confidence: "high" | "medium" | "low";
  httpHeaders?: Record<string, string>;
  lastSeen: number;

  // v0.2 新增 5 字段
  exePath?: string;            // /Applications/X.app/Contents/MacOS/x
  startedAt?: number;          // epoch ms,启动时间
  ppid?: number;               // 父进程 PID(后续可能用于更复杂分组)
  servicePreset?: {            // 端口预设匹配
    name: string;
    icon: string;               // icon 名字(UI 找对应 SVG/CSS)
    color: string;              // 主色
  };
  groupKey: string;            // 默认 = basename(exePath) || command
}

interface Preset {
  name: string;
  icon: string;
  color: string;
}
```

## 8. 错误处理(增量)

| 场景 | v0.1 | v0.2 增量 |
|---|---|---|
| `procinfo` 跨平台命令失败 | (无此模块) | 单 PID 失败降级为 undefined,不阻塞整次扫描 |
| `presets` 内置数据无匹配 | (无此模块) | 返回 undefined,UI 不显示徽章 |
| `theme.css` 加载失败 | (无) | 降级到深色 |
| 主题切换时组件未更新 | (无) | pub-sub 重新 render |
| 进程 exe path 权限不足 | (无) | catch 错误,exePath 留空 |
| 启动时间读取失败 | (无) | catch 错误,startedAt 留空 |

## 12. 风险与权衡

| 风险 | 缓解 |
|---|---|
| procinfo 跨平台命令在 macOS / Linux 行为差异 | 在测试里跑真实 spawn 验证;失败降级为 undefined |
| 2s 扫描 interval 内多端口采集 procinfo 慢(N 个端口 = N 个子命令) | 模块级 cache(同一 PID 不重复读);仅在 PID 首次出现时采集 |
| 前端 ES Modules 多文件加载慢 | 5 个文件总 < 50KB,HTTP/1.1 keep-alive 一次完成 |
| 主题 CSS variables 不被老浏览器支持 | 目标:最近 2 年版本 Chrome/Safari/Firefox/Edge;ES2020 baseline |
| 进程组规则单一(只看 exe basename) | 用户已接受;v0.3 可加 PPID 维度 |
| Dashboard 在 0 服务时显示空状态 | 显式空状态文案("No services running, start one to see it here") |

## 13. 实施切片(M1-M4)

v0.2 拆分 4 个独立 MVP 切片,每个可独立验收:

- **M1: 后端深度采集** —— 新增 `procinfo.ts`,`scanner.ts` 集成,`Service` 类型扩展,集成测试
- **M2: 端口预设库** —— 新增 `presets.ts`,`detector.ts` 集成,`/api/presets` 端点
- **M3: 前端组件拆分 + 现代 CSS** —— 拆 5 个 ES Modules,重写 `style.css`,引入 `theme.css`,主题切换
- **M4: Dashboard + 进程组 + 搜索** —— `dashboard.js` 组件,`services.js` 分组折叠,`filters.js` 搜索/筛选,响应式

每个 M 完成后跑 spec + code quality review,沿用 v0.1 subagent-driven 流程。
